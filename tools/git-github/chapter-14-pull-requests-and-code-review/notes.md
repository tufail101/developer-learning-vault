# Pull Requests And Code Review

## What This Chapter Is About

This chapter teaches how pull requests work and how review helps teams improve code before merging.
The goal is to help you use Git and GitHub with more confidence when the workflow gets a little more real.

## Real-World Analogy

A pull request is like handing a draft to another person before it gets printed in the final booklet. Review gives you a chance to catch confusion, bugs, and missing context early.

## Key Ideas

- A pull request proposes changes from one branch into another.
- A good PR description gives reviewers enough context to understand the change quickly.
- Code review is about improving the work, not winning an argument.

## Why This Matters

Git gets much easier when you stop treating commands like magic words.
You do not need every command at once, but you do need to know which problem each one is meant to solve.
That is what this chapter is helping you practice.

## Common Mistakes

- opening a huge pull request with unrelated changes mixed together
- writing no description and expecting reviewers to guess the intent
- treating review comments like personal attacks

## Practice Tip

Run commands in a throwaway repository first when the action feels risky.
That gives you room to learn without worrying about breaking something important.
