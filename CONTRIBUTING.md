# Contributing

This repository is designed as a long-term learning gift, so contributions should protect clarity, beginner-friendliness, and practical usefulness.

## How To Suggest A New Chapter

Open an issue or discussion that includes:

- the proposed chapter title
- where it fits in the learning path
- why it fills a real gap
- what a learner should be able to do after finishing it

Good chapter ideas usually:

- solve a real missing concept
- build naturally on earlier chapters
- include concrete examples and practice

## How To Report A Mistake

When reporting a mistake, include:

- the file path
- the incorrect or confusing section
- what you expected instead
- a suggested correction if you have one

Examples of useful reports:

- broken code example
- wrong explanation
- outdated command
- unclear beginner instruction
- missing accessibility or safety note

## Code Style Guidelines For Examples

- keep examples small and focused on the lesson
- prefer clear naming over clever naming
- use beginner-readable formatting
- add comments only when they genuinely reduce confusion
- avoid unnecessary abstraction in early chapters
- keep examples runnable whenever possible

For JavaScript examples:

- prefer modern syntax unless the lesson is about older syntax
- make side effects obvious
- avoid hiding core concepts inside helper utilities

For HTML and CSS examples:

- use semantic markup where appropriate
- keep selectors readable
- prefer realistic layout patterns over toy styling

## What Makes A Good Mini-Project

A good mini-project should:

- apply the chapter concepts in a realistic way
- have a clear goal
- be finishable by a learner in a reasonable amount of time
- include enough specificity that the learner knows what "done" looks like
- leave room for stretch improvements

A weak mini-project usually feels like:

- a vague prompt with no clear output
- a copy-paste version of the notes
- a task so large that the learner has no chance of finishing it

## Contribution Principles

- optimize for learner clarity first
- keep the tone encouraging and direct
- improve existing content before adding extra complexity
- prefer practical examples over abstract definitions
- do not remove good material just to rewrite in a different style

## Before Opening A PR

Please check:

- the content reads clearly for beginners
- file names and links still work
- code examples are formatted cleanly
- practice tasks are actionable
- mini-project instructions are specific enough to follow
